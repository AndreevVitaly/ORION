# Реестр решений

## DEC-0001 — Не использовать пиксели как основную единицу измерения

Дата: 2026
Статус: Active
Причина: пиксели зависят от масштаба, расстояния до камеры и размера изображения.
Альтернативы: абсолютные пиксельные расстояния, физическая калибровка в сантиметрах.
Последствия: измерения должны нормализоваться через внутренние геометрические отношения.
Связанные документы: `research/001_Origin.md`, `research/experiments/EXP-0001-IPD.md`

## DEC-0002 — Не принимать IPD как аксиому

Дата: 2026
Статус: Active
Причина: IPD является удобной классической базой, но в Profile его устойчивость должна быть проверена экспериментально.
Альтернативы: использовать IPD без проверки, выбрать другую базу заранее.
Последствия: IPD сравнивается с другими кандидатами через метрики стабильности.
Связанные документы: `research/000_Manifest.md`, `research/lic/LIC_Core.md`

## DEC-0003 — Разделить portrait_core и Dataset Builder

Дата: 2026
Статус: Active
Причина: `portrait_core` должен оставаться математическим ядром, а Dataset Builder — инструментом сбора данных.
Альтернативы: объединить сбор данных и расчеты в одном слое.
Последствия: архитектура сохраняет разделение анализа и подготовки датасетов.
Связанные документы: `research/dataset/DatasetBuilder.md`, `research/experiments/EXP-0004-Dataset-Builder.md`

## DEC-0004 — Создать research/ как научную память проекта

Дата: 2026
Статус: Active
Причина: проект развивается через гипотезы и эксперименты, а не только через код.
Альтернативы: хранить историю только в README или комментариях к коду.
Последствия: гипотезы, методология и решения фиксируются в отдельном исследовательском разделе.
Связанные документы: `research/README.md`, `research/000_Manifest.md`

## DEC-0005 — Хранить результаты анализа в JSON

Дата: 2026
Статус: Active
Причина: JSON удобен для воспроизводимых экспериментов, пакетной обработки и сравнения.
Альтернативы: хранить только визуальные результаты или свободный текст.
Последствия: эксперименты можно пересчитывать, сравнивать и агрегировать.
Связанные документы: `research/003_Methodology.md`, `research/experiments/EXP-0003-Landmark-Stability.md`

## DEC-0006 — Разделять кадры на passed/warning/rejected/duplicates

Дата: 2026
Статус: Active
Причина: качество кадра влияет на устойчивость landmark и должно анализироваться отдельно.
Альтернативы: смешивать все кадры в одной выборке.
Последствия: статистика становится более объяснимой и воспроизводимой.
Связанные документы: `research/dataset/QualityCriteria.md`, `research/dataset/DatasetRules.md`

## DEC-0007 — Не делать психологических выводов до построения надежной геометрической модели

Дата: 2026
Статус: Active
Причина: интерпретации без стабильной системы измерений могут быть недостоверными.
Альтернативы: преждевременно связывать геометрию лица с личностными характеристиками.
Последствия: Profile остается исследованием геометрии, landmark, LIC и воспроизводимости.
Связанные документы: `research/000_Manifest.md`, `research/003_Methodology.md`

## DEC-0008 — Введение стандарта PFR

Дата: 2026
Статус: Active
Причина: единый формат хранения геометрии лица для всей платформы Profile.
Альтернативы: сохранять разные варианты `portrait.json` в CLI, GUI, Dataset Builder и будущих приложениях.
Последствия: только `portrait_core` создает PFR, а приложения получают результат анализа через официальный API Scientific Engine.
Связанные документы: `docs/PFR_Specification.md`, `docs/architecture.md`

## DEC-0010 — Переход к модели научного архива

Дата: 2026
Статус: Active
Причина: Profile должен хранить не только отдельные JSON-файлы, а воспроизводимую цепочку исследования: PFR -> Dataset -> Experiment -> Report Pack -> Research.
Альтернативы: продолжать хранить результаты как набор несвязанных папок с JSON.
Последствия: вводятся Dataset Archive, Experiment identity, расширенная идентичность Report Pack и проверяемые связи между PFR и исследовательскими выводами.
Связанные документы: `docs/Archive_Model.md`, `docs/Dataset_Specification.md`, `docs/PFR_Specification.md`
